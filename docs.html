What is Orator?

Orator is an AI-powered code generator for Charli3 pull oracles on Cardano. Describe your oracle use case in plain English and get production-ready on-chain validators, off-chain integrations, and test suites in seconds.

The core idea Writing a Charli3 oracle integration from scratch takes days. You need to know Aiken or Plutus, understand the SDK, handle timestamp validation, fee estimation, and proof verification. Orator handles all of that from a single sentence.

Why it exists

Charli3 is Cardano's native oracle network. Integrating with it requires deep protocol knowledge, and most developers who want to build DeFi liquidators, parametric insurance contracts, or prediction markets get blocked at the oracle layer before they've written a line of application logic.

Orator removes that blocker. You describe what you want to build, and Orator generates the entire oracle integration, correctly structured with all the patterns Charli3 requires baked in.

What Orator is not

  • It is not a deployment tool. It generates code; you deploy it.
  • It is not a replacement for code review. Always read and test generated output before deploying to mainnet.
  • It is not a Charli3 node. It generates the client-side integration code that talks to the Charli3 network.

How It Works

Orator sends your use case description to a large language model (LLaMA 3.3 70B via Groq) with a detailed system prompt encoding the exact mechanics of Charli3 pull oracles. The API key is stored server-side. Your browser never touches it.

Request flow
Browser → POST /api/generate → Vercel serverless function
       → Groq API (LLaMA 3.3 70B)
       → Returns: 3 fenced code blocks
       → Browser renders with Copy buttons

What the AI knows about Charli3

The system prompt encodes these Charli3-specific facts so the generated code is always correct:

  • Pull model: contract submits an on-chain request; oracle nodes respond with signed data
  • Response payload: price/data + Unix timestamp + multi-node Ed25519 signature proof
  • Fee: 2 to 5 ADA per pull depending on freshness SLA
  • SDK: charli3-pull-oracle-sdk handles request building, fee estimation, response parsing
  • Staleness rule: reject oracle responses older than 60 seconds
  • Timeout pattern: if no oracle response within 5 minutes, trigger a refund path

Oracle Basics

Blockchains are intentionally isolated. A smart contract cannot fetch data from the internet. This is what makes them trustless and tamper-proof, but it creates a fundamental limitation: contracts cannot act on real-world events without a trusted data bridge.

An oracle is that bridge. It fetches real-world data (prices, weather, sports scores, flight statuses), verifies it through consensus, and posts it on-chain in a format smart contracts can read and trust.

Without oracles

  • A DeFi lending protocol cannot liquidate undercollateralised positions. It cannot know the asset price.
  • A stablecoin cannot maintain its peg. It cannot verify its own price.
  • A parametric insurance contract cannot pay out. It cannot verify the triggering event happened.
  • A prediction market cannot settle. It cannot verify the real-world outcome.

Oracles are what make any real-world connected DeFi primitive possible.

Charli3 Pull Model

Charli3 is Cardano's native oracle network. Unlike push oracles that post data on-chain continuously, Charli3 uses a pull model. Data is fetched on demand, when your contract requests it.

How a pull works

Pull oracle flow
1. Your contract submits an on-chain oracle request + fee (2–5 ADA)
2. Charli3 nodes detect the request
3. Each node fetches the data independently from multiple sources
4. Nodes sign their responses with Ed25519 keys
5. Nodes reach IQR consensus on the value
6. A signed aggregate response is posted on-chain
7. Your contract reads and verifies the proof
8. Your contract executes its logic based on the verified data

Key properties

  • On demand: you only pay when you actually need data
  • Multi-node signed: requires consensus from multiple independent nodes, so no single point of failure or manipulation
  • Timestamp bound: every response includes a Unix timestamp; stale data (over 60 seconds) should be rejected
  • Fee 2 to 5 ADA per pull: exact amount depends on the freshness SLA you request
  • Timeout safe: if no response within 5 minutes, your contract should provide a refund path

Trusting Generated Code

Short answer The same way you trust any AI-generated code. You read it, run the tests, and review it before deploying. Orator gives you a correct starting point, not a black box.

Why the output is trustworthy as a starting point

  • Fully transparent: every line is visible and readable. Nothing is hidden or compiled.
  • Grounded in Charli3 mechanics: the AI is given precise, verified facts about the protocol (fees, timestamp rules, proof format, SDK methods) before it generates anything.
  • Tests are included: a test suite ships with every generation specifically so you can verify behaviour before deployment.
  • Inline comments: every non-obvious decision in the code is explained so reviewers know why each pattern was used.

Your responsibilities before deploying

  • Read all three generated blocks and understand what each line does.
  • Run the test suite and ensure the mock oracle flow passes.
  • Deploy to Preprod first and exercise the full oracle request/response cycle on testnet.
  • Audit your business logic. Orator handles oracle plumbing; you are responsible for your contract's economic logic.
  • For high-value contracts, engage a Cardano smart contract auditor before mainnet.
Never deploy to mainnet without testing Oracle bugs can be catastrophic. Wrong price data can trigger mass liquidations, stale data can allow double-claims, and missing timeout paths can lock funds permanently. Always test end to end on Preprod first.

FAQ

Is Orator free to use? ▼
Yes. Orator is completely free. The underlying LLM (LLaMA 3.3 70B via Groq) is served through a serverless proxy. You don't need an API key or account to generate code.
What framework should I choose: Aiken or Plutus? ▼
Choose Aiken unless you have a specific reason not to. Aiken is purpose-built for Cardano, has a significantly friendlier syntax, compiles faster, and has better tooling for new projects. Plutus/Haskell is preferable only if your team is already working in an existing Haskell codebase.
Can I use the generated code commercially? ▼
Yes. There are no restrictions on how you use generated code. It's yours to modify, deploy, and build products on. Orator itself is MIT licensed.
The generated code has a bug. What do I do? ▼
Generate again with a more specific description. The more detail you provide about thresholds, edge cases, and conditions, the more precise the output. You can also modify the generated code directly; it's standard Aiken/TypeScript with comments explaining each section.
Does Orator actually submit transactions to Cardano? ▼
No. Orator generates code. It does not execute it. All on-chain activity happens when you take the generated code, deploy it yourself using Aiken and Mesh.js, and run your own transactions. Orator has no wallet integration and never touches your ADA.
What data feeds does Charli3 support? ▼
Charli3 supports a growing range of price pairs (ADA/USD, BTC/USD, ETH/USD, and more), as well as custom data feeds. Check the official Charli3 documentation for the full current list of supported feeds and their feed IDs.
Is my description sent anywhere other than Groq? ▼
Your input is sent to Groq's API (running LLaMA 3.3 70B) via a Vercel serverless function. It is not logged, stored, or used for training by Orator. Refer to Groq's privacy policy for their data handling terms.
